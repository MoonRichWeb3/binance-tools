# 05 AI 模块

本文记录 `binance-tools` 当前 AI 模块的设计、配置、桌面 UI 和请求流程。实现参考 Zed Agent 的交互方式，但按本项目需要做了轻量化实现。

## 代码位置

| 模块 | 路径 | 说明 |
| --- | --- | --- |
| 核心 AI 配置和请求 | `src/ai/mod.rs` | 读取 `config/ai.json`、选择 provider/model、发送普通和流式请求 |
| AI Chat 面板 | `examples/desktop-gpui/src/ui/ai/chat.rs` | 右侧 Agent 面板、模型选择、消息渲染、流式追加 |

| 标题栏入口 | `examples/desktop-gpui/src/ui/title_bar.rs` | 右侧 `AI` 按钮 |

## 配置文件

默认配置文件：

```text
config/ai.json
```

顶层结构：

| 字段 | 说明 |
| --- | --- |
| `agent` | Agent 行为配置，包括默认模型、输入框行数、参数规则等 |
| `language_models` | 内置 provider 和用户新增 OpenAI-compatible provider |
| `language_models.openai_compatible` | 用户在 Settings 里保存的 provider |

Provider 的主要字段：

| 字段 | 说明 |
| --- | --- |
| `api_url` | OpenAI-compatible base URL，例如 `https://api.example.com/v1` |
| `api_key` | 直接保存的 API Key |
| `api_key_env` | 从环境变量读取 API Key |
| `api_format` | 当前支持 `open_ai_chat`、`anthropic`、`google` |
| `available_models` | Provider 下可选模型列表 |

OpenAI-compatible URL 会自动拼接 `/chat/completions`。如果用户直接填完整 `/chat/completions` endpoint，代码会识别并避免重复拼接。

## Provider 选择规则

请求时使用当前 `ModelSelection { provider, model }` 查找 provider。

查找逻辑：

1. 优先在 `language_models.openai_compatible` 中按 provider 名查找。
2. 如果自定义 provider 下的 `available_models` 包含当前 model，则使用这个自定义 provider。
3. 如果自定义 provider 不匹配当前 model，再回退到内置 provider。

这样可以避免自定义 provider 和内置 provider 同名时，用错 URL 或 API Key。

建议：自定义 provider 避免命名为 `deepseek`、`openai` 这类内置名字，优先使用 `sensenova`、`z-ai`、`deepseek-ai` 这种明确名称。

## Settings 页面

AI Settings 页面包含：

- `External Agents`：展示 Codex CLI 占位信息。
- `Model Context Protocol (MCP) Servers`：MCP 占位区域。
- `LLM Providers`：展示用户保存的 OpenAI-compatible provider。

当前能力：

- `Configure` 从模型下拉进入 Settings。
- 顶部返回按钮关闭 Settings。
- 最大化按钮切换 Settings 是否覆盖主内容区。
- `Add Provider` 打开新增弹窗。
- 展开 provider 后显示 API URL、API Key Env、模型列表、配置文件位置。
- 展开 provider 后可点击 `修改` 和 `删除`。
- 保存 provider 后列表显示绿色勾。
- 新增/编辑时校验 Provider Name、API URL、API Key、Model Name 不能为空。
- 新增时 Provider Name 不能和现有 provider 重复。
- API Key 编辑时会去掉默认填充的长串 `0000`。

Provider 弹窗字段：

| 字段 | 说明 |
| --- | --- |
| Provider Name | provider key |
| API URL | OpenAI-compatible base URL |
| API Key | API Key |
| Model Name | 模型名，必须和服务商支持的模型名一致 |
| Max Completion Tokens | completion token 上限 |
| Max Output Tokens | 输出 token 上限 |
| Max Tokens | 上下文 token 上限 |
| Supports tools/images/parallel_tool_calls/prompt_cache_key | 能力声明，主要用于 UI 和后续扩展 |

## Chat 面板

右侧 AI Chat 面板当前能力：

- 标题栏右侧 `AI` 按钮打开/关闭。
- 空线程显示输入区和底部工具栏。
- 有消息后，上方为 thread，下方为固定 composer。
- composer 默认占约 `1/5` 高度，展开后占约 `4/5` 高度。
- Enter 发送消息。
- 发送后保留底部模型选择器和 toolbar。
- 模型下拉只展示 `config/ai.json` 中当前可用模型。
- 当前选中模型在下拉列表中显示绿色勾。
- `Configure` 固定在模型下拉底部。
- Context 图标 hover 显示输入/输出 token 估算。
- Assistant 回复支持 Markdown 渲染和文本选择复制。
- 回复下方提供点赞、点踩、复制、重试、继续按钮。

点赞/点踩当前只是本地 UI 状态，不会反馈给模型或服务端。

## 流式输出

OpenAI-compatible provider 使用流式输出：

```json
{
  "stream": true
}
```

请求入口：

```rust
send_chat_with_model_streaming_blocking(settings, selection, messages, on_delta)
```

实现方式：

1. UI 发送请求后先插入一条 `Thinking...` assistant 占位消息。
2. 后台线程请求 `/chat/completions` 并读取 SSE。
3. 每读到一段 `choices[0].delta.content`，通过 channel 发送给 UI。
4. UI 每隔约 `24ms` 轮询 delta，并追加到最后一条 assistant 消息。
5. 收到 `[DONE]` 后结束 loading。
6. 如果流式中途失败，保留已经输出的文本，并在末尾追加错误信息。

Anthropic 和 Google 当前仍走非流式回退：完整响应返回后一次性显示。

## 错误处理

AI HTTP 错误会统一转成中文提示：

| 状态 | 提示方向 |
| --- | --- |
| `401` | API Key 未授权、过期或环境变量未生效 |
| `403` | 权限、余额、区域或模型访问被拒绝 |
| `404` | 模型名或 API URL 不匹配 |
| `429` | 频率、额度或服务商限流 |
| `5xx` | 服务商临时错误，可重试 |
| 网络超时/连接失败 | 检查网络、代理、DNS 或本地服务 |

HTTP 客户端配置：

- connect timeout：`30s`
- request timeout：`300s`
- `reqwest` 使用 `rustls-tls-native-roots` 和 `system-proxy`，尽量贴近系统网络配置。

## Context 用量

底部 Context 图标显示本地估算值：

```text
Input: 7.7k / 168k
Output: 23 / 32k
```

说明：

- Input used：当前会话消息 + 输入框内容的本地估算。
- Input limit：当前模型 `max_tokens`。
- Output used：最近一条 assistant 回复的本地估算。
- Output limit：优先使用 `max_output_tokens`，没有时使用 `max_completion_tokens`。
- 当前不是官方 tokenizer，仅用于 UI 参考。

## 已知边界

- 工具调用、Web 搜索、MCP 目前仍是 UI/配置占位，没有真实执行链路。
- Anthropic/Google 尚未接入流式协议。
- Like/Dislike 未持久化。
- 多模型批量添加尚未做完整 UI。
